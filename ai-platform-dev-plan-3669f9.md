# AI训推一体数字孪生平台 — 系统开发规划

基于现有AI Studio训推一体平台，扩展为集**数据分层分级管理、数字孪生可视化、模型训推可视化、RAG调度决策**四大核心能力于一体的综合AI平台。

---

## 一、现有平台能力盘点

当前项目为 **ai-studio** （后端Java/Spring Cloud微服务）+ **ai-studio-front**（前端Vue 3 + Vite），已具备：

| 模块 | 功能 |
|---|---|
| **工作台首页** | 数据集/模型/训练任务/服务统计，集群节点CPU/GPU/内存监控 |
| **数据中心** | 数据集CRUD、多模态标注（图片分类、目标检测、文本、表格）、自动标注 |
| **模型中心** | 模型仓库、场景/框架筛选、版本管理、模型创建 |
| **模型产线** | DAG可视化编排、训练任务创建/执行/指标查看/打包 |
| **服务部署** | 模型一键部署为API服务、启停管理、日志查询 |
| **系统管理** | RBAC权限（用户/角色/部门/菜单/字典）|
| **系统监控** | 缓存/定时任务/登录日志/在线用户/操作日志/服务器状态 |

**技术栈：**
- 前端：Vue 3 + Vite + Element Plus + Ant Design Vue + ECharts + D3.js + dagre-d3 + TailwindCSS
- 后端：Spring Boot 2.3.2 + Spring Cloud (Nacos/Feign/Gateway) + MyBatis-Plus + MySQL + Redis
- 容器化：Docker + Kubernetes

## 三、模块一：数据分层分级管理

### 3.1 目标
对平台中的数据集、模型资产、训练日志等进行**分层（按来源/用途）** 和 **分级（按敏感度/安全等级）** 管理，实现数据资产的可视化治理。

### 3.2 功能清单
1. **数据资产目录**：自动扫描、注册平台内所有数据集、模型、日志等资产
2. **分类标签体系**：支持自定义多级分类标签（如：公开/内部/机密/绝密）
3. **分层管理**：按数据来源（原始数据/清洗数据/标注数据/特征数据）分层展示
4. **数据血缘追踪**：展示数据从采集→清洗→标注→训练→推理的全链路流转
5. **访问权限联动**：分级结果与RBAC权限系统联动，不同安全等级的数据对应不同访问权限
6. **数据资产看板**：可视化统计各层级、各等级数据分布

### 3.3 推荐开源项目

| 项目 | 用途 | GitHub Stars | 说明 |
|---|---|---|---|
| **OpenMetadata** | 数据目录 & 治理 | 6.2k+ | 现代化元数据管理平台，REST API优先，内置分类/标签/血缘功能，UI美观，部署简单 |
| **Apache Atlas** | 备选：数据治理 | 1.8k+ | Hadoop生态元数据治理，分类/血缘强大，但部署较重 |

### 3.4 开发方案
- **推荐方案：自研轻量模块（主）+ 参考OpenMetadata设计理念**
- 在现有后端新增 `data-governance` 微服务模块
- 前端新增 `views/dataGovernance/` 页面组，包含：资产目录、分级配置、血缘图（复用D3.js）、资产看板（复用ECharts）
- 核心表设计：`data_asset`（资产登记）、`data_classification`（分级定义）、`data_lineage`（血缘关系）、`asset_tag`（标签）
- 与现有数据集模块通过API打通，数据集创建时自动注册为数据资产

### 3.5 工期估算
- 后端数据模型 + API：**3-4天**
- 前端页面（资产目录/分级配置/血缘图/看板）：**5-6天**
- 与现有模块集成联调：**2天**
- **合计：约10-12天**

## 四、模块二：数字孪生可视化

### 4.1 目标
构建AI训推基础设施的**3D数字孪生**场景，实时映射集群节点、GPU卡、训练任务、推理服务的运行状态，实现"所见即所得"的态势感知。

### 4.2 功能清单
1. **3D机房/集群场景**：以3D方式展示服务器机柜、节点拓扑
2. **设备状态映射**：CPU/GPU/内存利用率实时映射到3D模型颜色/动画
3. **训练任务可视化**：训练任务以粒子流/动画形式在节点间流转
4. **推理服务可视化**：推理请求链路以光线/流线形式展示
5. **告警与态势感知**：异常节点高亮、告警弹窗、攻击态势展示
6. **交互能力**：点击设备查看详情面板，支持缩放/旋转/漫游

### 4.3 推荐开源项目

| 项目 | 用途 | GitHub Stars | 说明 |
|---|---|---|---|
| **TresJS** | Vue3 + Three.js 声明式框架 | 2.4k+ | Vue 3生态原生Three.js封装，组件式开发3D场景，与现有Vue3技术栈完美契合 |
| **three-vue-tres (tvt.js)** | 3D可视化底座 | 1.5k+ | 基于TresJS的三维可视化框架，内含数字孪生、数据中心等示例，国产化适配好 |
| **Three.js** | 底层3D渲染 | 104k+ | WebGL/WebGPU 3D引擎，生态最丰富 |
| **Babylon.js** | 备选3D引擎 | 23k+ | 微软出品，数字孪生IoT方案成熟，但与Vue集成不如TresJS自然 |

### 4.4 开发方案
- **推荐方案：TresJS + Three.js（主），参考 tvt.js 示例**
- 安装依赖：`@tresjs/core` + `three` + `@tresjs/cientos`（辅助工具集）
- 前端新增 `views/digitalTwin/` 页面组：
  - `DigitalTwinScene.vue`：3D主场景（机房/机柜/服务器模型）
  - `NodeDetailPanel.vue`：设备详情侧边栏
  - `AlertOverlay.vue`：告警叠加层
- 3D模型资产：使用Blender制作简化版服务器/机柜glTF模型，或使用Three.js几何体程序化生成
- 数据驱动：通过WebSocket/轮询获取集群实时数据（复用现有 `/monitor/server` API），映射到3D场景
- 后端新增 `digital-twin` 相关接口：场景配置、设备注册、实时状态推送（WebSocket）

### 4.5 工期估算
- 3D场景搭建（模型+场景+交互）：**8-10天**
- 数据驱动 + WebSocket实时同步：**3-4天**
- 态势感知 + 告警展示：**3-4天**
- 后端接口 + 场景配置：**3天**
- **合计：约17-21天**

## 六、模块四：RAG调度决策应用

### 6.1 目标
构建基于大模型+RAG的**智能调度决策系统**，当平台需要执行训练/推理任务调度时，大模型会检索RAG知识库中的调度策略、运维经验、最佳实践，给出决策建议和操作步骤。

### 6.2 功能清单
1. **RAG知识库管理**：上传/管理调度策略文档、运维手册、最佳实践文档
2. **知识库向量化**：文档自动分块、Embedding、存入向量数据库
3. **对话式调度助手**：聊天界面，用户描述需求，大模型检索知识后给出调度决策
4. **调度指令生成**：根据集群状态+知识库，自动生成训练任务调度建议（如：推荐节点、资源配额、队列优先级）
5. **决策审计日志**：记录每次决策的检索上下文、推理过程、最终结果
6. **与平台联动**：决策结果可一键应用到训练任务/服务部署

### 6.3 推荐开源项目

| 项目 | 用途 | GitHub Stars | 说明 |
|---|---|---|---|
| **Dify** | RAG + Agent + Workflow 平台 | 75k+ | 开源LLM应用平台，**强烈推荐**。内置RAG管线、可视化Workflow、Agent框架、多模型对接，支持Docker自部署，REST API完善，可直接作为RAG后端 |
| **RAGFlow** | 备选RAG引擎 | 35k+ | 文档处理能力强，深度文档解析+可视化引用溯源 |
| **LangChain** | 备选RAG框架 | 100k+ | 灵活但需要更多自研工作，适合定制化需求 |
| **Milvus** | 向量数据库 | 32k+ | 高性能向量数据库，Dify默认集成 |
| **Ollama** | 本地LLM部署 | 120k+ | 本地运行开源大模型（Qwen/Llama/DeepSeek等），与Dify无缝集成 |

### 6.4 开发方案
- **推荐方案：Dify（RAG+Agent后端）+ Ollama（本地LLM）+ 前端自研对话界面**
- **第一步**：Docker Compose部署Dify + Ollama
  - Dify提供知识库管理、RAG管线、Workflow编排能力
  - Ollama运行开源大模型（如Qwen2.5、DeepSeek-V3等）
- **第二步**：在Dify中创建"调度决策Agent"
  - 上传调度策略文档到知识库
  - 配置Workflow：接收用户问题 → 检索知识库 → 结合集群状态（通过API Tool调用平台接口）→ 生成决策
- **第三步**：前端集成
  - 新增 `views/ragScheduler/` 页面组：
    - `ChatAssistant.vue`：对话式调度助手（调用Dify Chat API）
    - `KnowledgeBase.vue`：知识库管理（调用Dify Knowledge API）
    - `DecisionLog.vue`：决策审计日志
  - 在训练任务创建页面增加"AI推荐调度"按钮
- **第四步**：后端对接
  - 新增 `rag-scheduler` 微服务或在现有服务中增加Dify API对接层
  - 实现：调度决策请求 → Dify API → 返回决策结果 → 记录审计日志

### 6.5 工期估算
- Dify + Ollama 部署配置：**2-3天**
- 知识库整理 + Agent Workflow配置：**3-4天**
- 前端对话界面 + 知识库管理页面：**5-6天**
- 后端API对接层 + 审计日志：**3-4天**
- 与训练/推理模块联动：**2-3天**
- **合计：约15-20天**

## 十、技术风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 3D场景性能瓶颈 | 大量节点时帧率下降 | LOD分层渲染、实例化渲染、按需加载 |
| LLM推理延迟高 | 调度决策响应慢 | 使用量化模型(Q4)、流式输出、缓存常见决策 |
| Dify版本升级不兼容 | API变更 | 封装Adapter层隔离、锁定版本 |
| 数据血缘关系复杂 | 展示混乱 | 采用分层/分区布局、支持搜索过滤 |
| GPU资源不足 | 无法同时训练+推理LLM | Ollama支持CPU推理备选、使用小参数模型 |
