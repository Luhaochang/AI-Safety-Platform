# 3.2.2 系统功能设计与实现 —— 类图设计说明

> 本文档对应论文第三章 3.2.2 节，详细说明知识库管理、调度助手和决策审计三个功能模块的核心类图设计。
> PlantUML 源文件位于 `docs/class_diagram_knowledge.puml`、`docs/class_diagram_chat.puml`、`docs/class_diagram_audit.puml`。

## 3.2.2.1 知识库管理核心类图（图3-10）

### 类图概述

知识库管理模块以知识库为基本组织单元，对文档对象、知识片段和处理任务进行统一管理。该模块的核心类图包含 **8个类**，分别为知识库实体类（KnowledgeBase）、文档实体类（Document）、知识片段实体类（Chunk）、文本提取器（TextExtractor）、文本清洗器（TextCleaner）、文本分块器（TextChunker）、向量存储管理器（VectorStore）和元数据管理器（MetadataManager），以及辅助数据类清洗结果（CleanResult）。

### 核心类说明

#### （1）KnowledgeBase（知识库实体类）

知识库是知识管理的顶层组织单元，用于限定知识边界与检索范围。每个知识库对应向量数据库中的一个独立集合。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | str | 知识库唯一标识 |
| name | str | 知识库名称 |
| description | str | 知识库描述 |
| chunk_count | int | 包含的知识片段总数 |
| document_count | int | 包含的文档数量 |
| word_count | int | 总字数 |
| create_time | str | 创建时间 |
| update_time | str | 最后更新时间 |

| 方法 | 说明 |
|------|------|
| to_dict() | 将对象序列化为字典，用于API响应 |
| from_dict(kb_id, data) | 从元数据字典构造知识库对象 |

#### （2）Document（文档实体类）

表示上传至知识库的原始文档对象，记录文档的基本属性与处理状态。一个文档经处理后产生多个Chunk。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | str | 文档唯一标识 |
| name | str | 文档文件名 |
| kb_id | str | 所属知识库ID |
| file_path | str | 文件存储路径 |
| size | str | 文件大小 |
| status | str | 处理状态（待处理/已索引） |
| chunk_count | int | 切分产生的片段数 |
| word_count | int | 文档字数 |
| chunk_size | int | 切分块大小配置 |
| chunk_overlap | int | 切分重叠字符数配置 |
| create_time | str | 上传时间 |

| 方法 | 说明 |
|------|------|
| to_dict() | 序列化为API响应字典 |
| from_dict(doc_id, data) | 从元数据字典构造文档对象 |

#### （3）Chunk（知识片段实体类）

经文本切分后的最小知识单元，是向量检索的基本粒度。每个Chunk携带文本内容、向量表示、位置信息及来源元数据。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | str | 片段唯一标识 |
| text | str | 文本内容 |
| doc_id | str | 所属文档ID |
| doc_name | str | 所属文档名 |
| chunk_index | int | 在文档中的序号 |
| start / end | int | 在原文中的起止字符位置 |
| char_count | int | 字符数 |
| embedding | List[float] | 向量表示 |
| embedding_dim | int | 向量维度 |
| metadata | dict | 附加元数据 |

#### （4）TextExtractor（文本提取器）

负责从不同格式文档中提取纯文本，支持PDF、Word和纯文本格式。

| 方法 | 说明 |
|------|------|
| extract(file_path, filename) | 根据文件后缀自动选择提取方式 |
| extract_from_pdf(file_path) | 从PDF文件提取文本（pdfplumber/PyPDF2） |
| extract_from_docx(file_path) | 从Word文件提取文本（python-docx） |
| extract_from_txt(file_path) | 从纯文本文件提取（自动编码检测） |

#### （5）TextCleaner（文本清洗器）

对原始文本执行多种清洗规则，降低噪声干扰，提高知识质量。

| 属性 | 类型 | 说明 |
|------|------|------|
| SUPPORTED_RULES | List[str] | 支持的清洗规则列表 |

| 方法 | 说明 |
|------|------|
| clean(text, rules) | 按规则序列执行清洗，返回CleanResult |
| _rule_html_tags(text) | 去除HTML标签和实体引用 |
| _rule_extra_whitespace(text) | 压缩多余空白字符和空行 |
| _rule_phone_mask(text) | 手机号码脱敏处理 |
| _rule_id_mask(text) | 身份证号码脱敏处理 |
| _rule_email_mask(text) | 电子邮箱脱敏处理 |
| _rule_special_chars(text) | 去除零宽字符等特殊字符 |
| _rule_dedup_lines(text) | 去除重复行 |

#### （6）TextChunker（文本分块器）

将长文本按滑动窗口策略切分为适于语义检索的知识片段。

| 属性 | 类型 | 说明 |
|------|------|------|
| chunk_size | int | 每个片段的字符数上限 |
| chunk_overlap | int | 相邻片段的重叠字符数 |

| 方法 | 说明 |
|------|------|
| chunk(text) | 使用实例配置执行分块 |
| sliding_window_chunk(text, size, overlap) | 滑动窗口分块算法 |

#### （7）VectorStore（向量存储管理器）

封装ChromaDB向量数据库操作，负责向量集合管理和知识片段的存储与检索。

| 属性 | 类型 | 说明 |
|------|------|------|
| chroma_client | PersistentClient | ChromaDB持久化客户端 |
| embedding_function | EmbeddingFunction | SentenceTransformer嵌入函数 |

| 方法 | 说明 |
|------|------|
| create_collection(kb_id, metadata) | 创建或获取向量集合 |
| delete_collection(kb_id) | 删除向量集合 |
| add_chunks(kb_id, ids, documents, metadatas) | 批量添加知识片段 |
| delete_chunks_by_doc(kb_id, doc_id) | 按文档ID删除片段 |
| get_chunks(kb_id, doc_id, page, page_size) | 分页查询片段详情（含向量） |
| query(kb_id, query_text, top_k) | 语义相似度检索 |

#### （8）MetadataManager（元数据管理器）

负责知识库和文档元数据的JSON文件持久化存储与读写。

| 方法 | 说明 |
|------|------|
| load() / save(meta) | 加载/保存元数据文件 |
| list_knowledge_bases() | 列出所有知识库 |
| list_documents(kb_id) | 列出知识库下所有文档 |
| register_knowledge_base(kb) | 注册新知识库 |
| remove_knowledge_base(kb_id) | 删除知识库及关联文档 |
| register_document(doc) | 注册新文档 |
| remove_document(doc_id) | 删除文档 |

### 类间关系

| 关系 | 说明 |
|------|------|
| KnowledgeBase 1 ──* Document | 组合关系，一个知识库包含多个文档 |
| Document 1 ──* Chunk | 组合关系，一个文档切分为多个知识片段 |
| Document ···> TextExtractor | 依赖关系，文档使用文本提取器提取文本 |
| Document ···> TextCleaner | 依赖关系，文档使用文本清洗器清洗内容 |
| Document ···> TextChunker | 依赖关系，文档使用文本分块器执行切分 |
| TextCleaner ···> CleanResult | 依赖关系，清洗器返回清洗结果对象 |
| Chunk ──> VectorStore | 关联关系，知识片段向量化后存入向量库 |
| KnowledgeBase ──> VectorStore | 关联关系，知识库对应向量库中的集合 |
| KnowledgeBase ──> MetadataManager | 关联关系，知识库元数据由管理器持久化 |
| Document ──> MetadataManager | 关联关系，文档元数据由管理器持久化 |

## 3.2.2.2 调度助手核心类图（图3-13）

### 类图概述

调度助手模块围绕会话对象、检索对象、生成对象和回答对象展开设计。该模块的核心类图包含 **9个类**，分别为会话实体类（Conversation）、消息实体类（Message）、引用来源类（Reference）、检索器（Retriever）、重排器（Reranker）、大语言模型生成器（LLMGenerator）、RAG流水线（RAGPipeline）、生成结果类（RAGResult）和会话管理器（ConversationManager）。其中会话对象用于维护历史对话上下文，检索器用于组织问题表达与知识召回，重排器用于对检索结果进行精排序，生成器用于完成增强提示构造与模型调用，回答对象则负责组织生成结果及其引用内容，RAG流水线协调上述组件形成完整的对话处理链路。

### 核心类说明

#### （1）Conversation（会话实体类）

维护一轮完整的对话上下文，是调度助手的核心交互载体。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | str | 会话唯一标识 |
| title | str | 会话标题（取自首条问题） |
| user | str | 用户标识 |
| create_time | str | 创建时间 |
| update_time | str | 最后更新时间 |
| messages | List[Message] | 消息序列 |

| 方法 | 说明 |
|------|------|
| add_message(message) | 追加消息并自动更新标题和时间 |
| to_dict() | 序列化为字典 |
| from_dict(data) | 从字典构造会话对象 |

#### （2）Message（消息实体类）

表示一条对话消息，包括用户问题或助手回答。助手消息可关联检索引用来源。

| 属性 | 类型 | 说明 |
|------|------|------|
| role | str | 角色（user/assistant） |
| content | str | 消息文本内容 |
| time | str | 发送时间 |
| references | List[Reference] | 关联的引用来源列表 |

#### （3）Reference（引用来源类）

表示RAG检索过程中召回的知识片段引用，用于结果溯源和可解释展示。

| 属性 | 类型 | 说明 |
|------|------|------|
| title | str | 来源文档名称 |
| section | str | 片段位置标识 |
| relevance | float | 相关性评分 |
| text | str | 引用文本摘要 |

#### （4）Retriever（检索器）

实现多策略知识检索，包括基于BM25的稀疏检索、基于SentenceTransformer的稠密检索和基于RRF的混合检索。

| 属性 | 类型 | 说明 |
|------|------|------|
| sentence_model | SentenceTransformer | 句向量编码模型 |
| vector_store | VectorStore | 向量存储引用 |
| bm25_available | bool | BM25是否可用 |

| 方法 | 说明 |
|------|------|
| dense_retrieve(query, kb_ids, top_k) | 基于向量相似度的稠密检索 |
| bm25_retrieve(query, documents, top_k) | 基于词频统计的BM25稀疏检索 |
| hybrid_retrieve(query, kb_ids, top_k) | 融合Dense和BM25的混合检索 |
| _rrf_fusion(dense, bm25, top_k) | 倒数排名融合（RRF）算法 |

#### （5）Reranker（重排器）

基于CrossEncoder模型对候选检索结果进行精细化重排序，通过交叉编码计算查询与候选片段的语义相关性评分。

| 属性 | 类型 | 说明 |
|------|------|------|
| cross_encoder | CrossEncoder | 交叉编码器模型 |

| 方法 | 说明 |
|------|------|
| rerank(query, documents, top_k) | 对候选文档执行CrossEncoder重排序 |

#### （6）LLMGenerator（大语言模型生成器）

封装通义千问Qwen API调用逻辑，负责增强提示构造和大语言模型生成调用。

| 属性 | 类型 | 说明 |
|------|------|------|
| api_key | str | API密钥 |
| api_url | str | API端点URL |
| default_model | str | 默认模型名称 |
| available | bool | API连接状态 |
| SYSTEM_PROMPT | str | 系统角色提示词 |

| 方法 | 说明 |
|------|------|
| generate(prompt, model) | 调用LLM生成回答 |
| build_rag_prompt(query, contexts) | 构造检索增强提示模板 |
| check_connection() | 检测API连接可用性 |

#### （7）RAGPipeline（RAG流水线）

编排完整的检索增强生成流程，协调Retriever、Reranker和LLMGenerator三个组件。

| 属性 | 类型 | 说明 |
|------|------|------|
| retriever | Retriever | 检索器实例 |
| reranker | Reranker | 重排器实例 |
| generator | LLMGenerator | 生成器实例 |
| metadata_mgr | MetadataManager | 元数据管理器 |

| 方法 | 说明 |
|------|------|
| query(query, kb_ids, top_k, model) | 执行完整RAG流程并返回结果 |
| _build_fallback_answer(reranked) | API不可用时构建摘要回答 |

#### （8）RAGResult（生成结果类）

封装完整RAG流程的输出，包括生成回答、上下文片段和引用来源。

| 属性 | 类型 | 说明 |
|------|------|------|
| answer | str | LLM生成的回答文本 |
| contexts | List[dict] | 重排后的上下文片段 |
| initial_docs | List[dict] | 初始检索文档 |
| references | List[Reference] | 引用来源列表 |

#### （9）ConversationManager（会话管理器）

负责对话历史的持久化存储与管理。

| 方法 | 说明 |
|------|------|
| load_all() | 加载所有对话历史 |
| get_conversation(conv_id) | 获取单个对话详情 |
| save_message(conv_id, query, answer, refs) | 保存问答消息到对话 |
| delete_conversation(conv_id) | 删除指定对话 |

### 类间关系

| 关系 | 说明 |
|------|------|
| Conversation 1 ──* Message | 组合关系，会话包含多条消息 |
| Message 1 ○──* Reference | 聚合关系，消息可关联多个引用来源 |
| RAGPipeline ○── Retriever | 组合关系，流水线持有检索器 |
| RAGPipeline ○── Reranker | 组合关系，流水线持有重排器 |
| RAGPipeline ○── LLMGenerator | 组合关系，流水线持有生成器 |
| RAGPipeline ···> RAGResult | 依赖关系，流水线输出生成结果 |
| RAGResult ○──* Reference | 聚合关系，结果包含引用来源 |
| ConversationManager ──> Conversation | 关联关系，管理器管理会话对象 |
| Retriever ···> VectorStore | 依赖关系，检索器调用向量库执行检索 |

## 3.2.2.3 决策审计核心类图（图3-15）

### 类图概述

决策审计模块围绕日志对象、问答记录对象和导出对象进行设计。该模块的核心类图包含 **8个类**，分别为决策日志基类（DecisionLog）、对话决策日志（ChatDecisionLog）、推演决策日志（TwinDecisionLog）、检索语段类（RetrievedPassage）、推演结果类（TwinResult）、日志管理器（DecisionLogManager）、日志查询服务（LogQueryService）和日志导出服务（LogExportService）。其中日志基类定义了问答过程的共性字段，对话日志和推演日志通过继承扩展各自的专属信息，日志管理器负责日志的统一生成与查询，导出服务则支撑日志的格式化输出与归档。

### 核心类说明

#### （1）DecisionLog（决策日志基类）

记录单次问答交互的关键信息，是决策审计的基本数据单元。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | str | 日志唯一标识 |
| type | str | 日志类型（chat/twin） |
| type_label | str | 类型显示标签 |
| question | str | 问题或场景描述 |
| answer | str | 回答或结论摘要 |
| user | str | 执行用户 |
| duration | str | 响应耗时 |
| create_time | str | 执行时间 |

| 方法 | 说明 |
|------|------|
| to_dict() | 序列化为字典 |

#### （2）ChatDecisionLog（对话决策日志）

继承自DecisionLog，额外记录完整消息序列和检索语段，用于对话过程的完整审计。

| 新增属性 | 类型 | 说明 |
|------|------|------|
| messages | List[Message] | 完整对话消息列表 |
| retrieved_passages | List[RetrievedPassage] | 检索召回的知识语段 |

| 方法 | 说明 |
|------|------|
| from_conversation(conv) | 从会话对象生成对话日志 |
| to_dict() | 序列化（含消息和检索语段） |

#### （3）TwinDecisionLog（推演决策日志）

继承自DecisionLog，额外记录推演场景、配置参数和仿真结果，用于推演过程的审计追溯。

| 新增属性 | 类型 | 说明 |
|------|------|------|
| scenario | str | 推演场景描述 |
| params | dict | 推演参数配置 |
| twin_result | TwinResult | 推演仿真结果 |
| risk_level | str | 风险等级评估 |

#### （4）RetrievedPassage（检索语段）

记录RAG检索召回的知识片段详情，用于证据追溯与质量评估。

| 属性 | 类型 | 说明 |
|------|------|------|
| doc_title | str | 来源文档标题 |
| section | str | 片段位置标识 |
| relevance | float | 相关性评分 |
| content | str | 语段文本内容 |

#### （5）TwinResult（推演结果）

封装孪生推演的仿真输出，包含推演结论、影响设备分析和处置建议。

| 属性 | 类型 | 说明 |
|------|------|------|
| summary | str | 推演结论摘要 |
| affected_lines | List[dict] | 受影响设备/线路列表 |
| recommendations | List[str] | 建议措施列表 |
| risk_level | str | 风险等级 |

#### （6）DecisionLogManager（日志管理器）

从对话历史和推演记录中生成结构化决策日志，提供统一查询与统计。

| 属性 | 类型 | 说明 |
|------|------|------|
| conversation_mgr | ConversationManager | 会话管理器引用 |

| 方法 | 说明 |
|------|------|
| get_all_logs() | 获取全部决策日志 |
| search_logs(keyword, type_filter) | 按关键词和类型筛选日志 |
| get_log_detail(log_id) | 获取单条日志详情 |
| get_statistics() | 获取日志统计信息 |

#### （7）LogQueryService（日志查询服务）

提供灵活的日志检索与筛选能力，支持关键词搜索、类型过滤和条件重置。

| 属性 | 类型 | 说明 |
|------|------|------|
| keyword | str | 搜索关键词 |
| type_filter | str | 类型过滤条件 |
| date_range | tuple | 时间范围筛选 |

| 方法 | 说明 |
|------|------|
| search(keyword) | 关键词搜索 |
| filter_by_type(type) | 按类型筛选 |
| reset() | 重置所有筛选条件 |

#### （8）LogExportService（日志导出服务）

支持将决策日志导出为CSV等格式，便于离线复查、归档和管理。

| 方法 | 说明 |
|------|------|
| export_to_csv(logs) | 导出日志为CSV格式 |
| format_log_data(log) | 格式化单条日志数据 |

### 类间关系

| 关系 | 说明 |
|------|------|
| DecisionLog <\|── ChatDecisionLog | 继承关系，对话日志扩展基类 |
| DecisionLog <\|── TwinDecisionLog | 继承关系，推演日志扩展基类 |
| ChatDecisionLog 1 ○──* RetrievedPassage | 聚合关系，对话日志包含检索语段 |
| TwinDecisionLog 1 *── 1 TwinResult | 组合关系，推演日志包含推演结果 |
| DecisionLogManager ···> DecisionLog | 依赖关系，管理器查询和管理日志 |
| DecisionLogManager ···> ChatDecisionLog | 依赖关系，从会话生成对话日志 |
| DecisionLogManager ···> TwinDecisionLog | 依赖关系，从推演生成推演日志 |
| LogExportService ···> DecisionLog | 依赖关系，格式化和导出日志 |
| LogQueryService ···> DecisionLog | 依赖关系，检索和筛选日志 |

## 论文写作参考文字

### 图3-10 描述段落（知识库管理）

> 图3-10为知识库管理核心类图。该模块以KnowledgeBase类为顶层组织单元，对Document、Chunk和处理任务类进行统一管理。KnowledgeBase与Document之间为一对多的组合关系，一个知识库包含多个文档；Document与Chunk之间同样为一对多关系，一个文档经切分后产生多个知识片段。在知识处理流程方面，Document依赖TextExtractor完成文本提取，依赖TextCleaner执行文本清洗（返回CleanResult），依赖TextChunker执行文本分块。生成的Chunk通过VectorStore写入ChromaDB向量数据库，KnowledgeBase和Document的注册信息则由MetadataManager负责持久化管理。

### 图3-13 描述段落（调度助手）

> 图3-13为调度助手核心类图。该模块围绕Conversation（会话）、Retriever（检索器）、Reranker（重排器）、LLMGenerator（生成器）和RAGResult（回答结果）展开设计。Conversation与Message之间为一对多的组合关系，维护完整的对话上下文；Message可聚合多个Reference，记录回答的知识来源。在RAG处理链路方面，RAGPipeline以组合方式持有Retriever、Reranker和LLMGenerator三个核心组件：Retriever负责执行BM25稀疏检索、Dense稠密检索和基于RRF的混合检索，从向量库中召回候选知识片段；Reranker通过CrossEncoder模型对候选结果进行精细化重排序；LLMGenerator构造增强提示并调用通义千问大语言模型生成回答。RAGPipeline协调上述组件执行完整的检索增强生成流程，输出RAGResult对象。ConversationManager负责会话的持久化存储与管理。

### 图3-15 描述段落（决策审计）

> 图3-15为决策审计核心类图。该模块以DecisionLog基类为核心，通过继承机制派生出ChatDecisionLog（对话日志）和TwinDecisionLog（推演日志）两个子类。DecisionLog定义了日志类型、问题内容、回答结果、用户信息、响应耗时和执行时间等共性字段；ChatDecisionLog额外关联完整的消息序列和RetrievedPassage检索语段列表，支持对话过程的完整复查；TwinDecisionLog额外包含推演场景、参数配置和TwinResult推演结果（含结论摘要、影响设备分析和建议措施），支持推演过程的分析追溯。在管理服务方面，DecisionLogManager负责从对话历史和推演记录中统一生成结构化日志并提供查询与统计功能；LogQueryService提供关键词搜索、类型筛选和条件重置等灵活的检索能力；LogExportService支持将日志导出为CSV等格式，便于离线归档和统一管理。
