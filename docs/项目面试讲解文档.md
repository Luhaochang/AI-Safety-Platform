# 电力调度AI决策平台 - 项目面试讲解文档

## 一、项目概述

### 1.1 项目背景
本项目是一个面向电力调度领域的智能决策辅助平台，基于RAG（检索增强生成）技术，结合知识库管理、智能对话、数字孪生推演等功能，为电力调度人员提供专业的决策支持服务。

### 1.2 技术架构
- **前端**: Vue 3 + Element Plus + Pinia + Vue Router
- **后端**: FastAPI (Python) + ChromaDB + SentenceTransformers
- **AI能力**: 通义千问Qwen大模型 + RAG检索增强生成
- **部署**: 前后端分离架构，支持本地/云端部署

### 1.3 核心价值
- 将海量电力调度规程、案例文档转化为可检索的知识库
- 通过RAG技术实现有据可依的智能问答
- 记录决策全过程，支持事后审计与复盘
- 数字孪生推演辅助风险预判

### 2.1 知识库管理模块

#### 功能描述
负责电力调度知识的采集、清洗、分块、向量化存储，构建可检索的知识库体系。

#### 核心功能
1. **知识库CRUD**
   - 创建知识库（支持分类管理）
   - 查看知识库列表及统计信息（文档数、块数、字数）
   - 删除知识库及关联数据

2. **文档处理流水线**
   - 支持格式：PDF、Word（.docx）、纯文本（.txt/.md）
   - 文本提取：pdfplumber/PyPDF2（PDF）、python-docx（Word）、chardet编码检测（文本）
   - 文本清洗：HTML标签去除、空白压缩、敏感信息脱敏（手机号/身份证/邮箱）
   - 文本分块：滑动窗口策略（chunk_size=512, chunk_overlap=64）
   - 向量化：SentenceTransformer（all-MiniLM-L6-v2）生成384维向量
   - 持久化：ChromaDB向量数据库

3. **知识检索**
   - 语义检索：基于向量相似度的Dense检索
   - 分页查看：支持查看文档的所有Chunk及向量预览
   - 元数据管理：JSON文件持久化知识库和文档元信息

### 2.2 调度助手模块（RAG对话）

#### 功能描述
基于RAG技术的智能对话系统，从知识库检索相关内容后，调用大模型生成专业回答。

#### 核心功能
1. **多轮对话管理**
   - 会话创建与历史记录
   - 消息持久化（JSON文件）
   - 会话列表与切换

2. **RAG检索增强生成流程**
   - **混合检索**: BM25（稀疏）+ Dense（稠密）+ RRF融合
   - **重排序**: CrossEncoder精细化相关性评分
   - **LLM生成**: 通义千问Qwen API生成专业回答
   - **引用溯源**: 展示检索来源（文档标题、章节、相关度）

3. **智能降级**
   - RAG服务未连接时，直接调用Qwen API
   - 显示连接状态（RAG/Qwen）

**后端（FastAPI + RAG Pipeline）**
```python
# services.py - RAG流水线
class RAGPipeline:
    def query(self, query: str, kb_ids: List[str]):
        # 1. 混合检索（BM25 + Dense + RRF融合）
        retrieved = self.retriever.hybrid_retrieve(query, kb_ids, top_k=10)
        
        # 2. CrossEncoder重排
        reranked = self.reranker.rerank(query, retrieved, top_k=3)
        
        # 3. 构建增强提示
        prompt = self.generator.build_rag_prompt(query, reranked)
        
        # 4. Qwen生成回答
        answer = self.generator.generate(prompt, model)
        
        return RAGResult(answer=answer, references=references)
```

#### 关键技术点
- **混合检索**: BM25捕捉关键词，Dense捕捉语义，RRF融合取长补短
- **CrossEncoder**: 比向量检索更精准的相关性评分（query-doc交叉编码）
- **提示工程**: 将检索上下文注入Prompt，引导模型基于知识回答
- **Pinia状态管理**: 集中管理会话历史，支持跨组件共享

## 四、核心技术实现亮点

### 4.1 RAG检索增强生成

#### 混合检索策略
```python
def hybrid_retrieve(self, query: str, kb_ids: List[str], top_k: int = 5):
    # 1. Dense检索（向量相似度）
    dense_results = self.dense_retrieve(query, kb_ids, top_k * 3)
    
    # 2. BM25检索（关键词匹配）
    bm25_results = self.bm25_retrieve(query, [r["text"] for r in dense_results], top_k * 3)
    
    # 3. RRF融合（Reciprocal Rank Fusion）
    return self._rrf_fusion(dense_results, bm25_results, top_k)
```

**优势**:
- Dense检索捕捉语义相似性（"故障"≈"异常"）
- BM25检索捕捉精确关键词（"220kV"必须匹配）
- RRF融合平衡两者，提升召回率和准确率

#### CrossEncoder重排
```python
def rerank(self, query: str, documents: List[dict], top_k: int = 5):
    # 构造query-doc对
    pairs = [[query, doc["text"]] for doc in documents]
    
    # CrossEncoder打分（0-1之间）
    scores = self.cross_encoder.predict(pairs)
    
    # 按分数排序，取Top-K
    scored = sorted(zip(documents, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, score in scored[:top_k]]
```

**优势**:
- 比向量检索更精准（query和doc同时编码，捕捉细粒度交互）
- 计算成本高，只对候选集重排（先召回再精排）

export const useChatStore = defineStore('chat', {

    }

      const conv = this.currentConversation;
      if (conv) {
        conv.messages.push(message);
        conv.updateTime = new Date().toISOString();
      }
    }
  }
});

## 六、面试常见问题

### 6.1 技术选型问题

**Q: 为什么选择FastAPI而不是Flask/Django？**

A: 
1. **性能**: FastAPI基于ASGI，支持异步，性能优于Flask（WSGI）
2. **自动文档**: 自动生成OpenAPI/Swagger文档，前后端联调方便
3. **类型检查**: 基于Pydantic，参数自动验证，减少bug
4. **现代化**: 原生支持async/await，适合AI推理等IO密集型任务

**Q: 为什么选择ChromaDB而不是Milvus/Pinecone？**

A:
1. **轻量级**: 无需额外服务，本地持久化，部署简单
2. **开箱即用**: 集成SentenceTransformer，无需手动管理向量
3. **适合场景**: 中小规模知识库（<100万条），满足电力调度场景需求
4. **成本**: 完全免费，无云服务费用

**Q: 为什么选择Vue 3而不是React？**

A:
1. **学习曲线**: Vue模板语法更接近HTML，上手快
2. **组合式API**: 逻辑复用能力强，代码组织清晰
3. **生态**: Element Plus成熟稳定，适合企业后台
4. **性能**: Vue 3响应式系统基于Proxy，性能优于Vue 2

---

### 6.2 架构设计问题

**Q: RAG流程中为什么要先召回再重排？**

A:
1. **性能平衡**: Dense检索速度快（向量相似度计算），适合大规模召回
2. **精度提升**: CrossEncoder精度高但慢，只对候选集重排（10→3）
3. **两阶段检索**: 召回保证覆盖率，重排保证准确率

**Q: 如何保证对话历史的一致性？**

A:
1. **前端**: Pinia状态管理，单一数据源
2. **后端**: JSON文件持久化，每次对话后立即保存
3. **同步**: 前端定期从后端拉取最新历史（避免多端不一致）

**Q: 如何处理大文件上传？**

A:
1. **分块上传**: 前端使用FormData，后端FastAPI自动处理multipart
2. **异步处理**: 文档处理（提取→分块→向量化）在后台异步执行
3. **进度反馈**: WebSocket推送处理进度（可选）

---

### 6.3 性能优化问题

**Q: 如何优化向量检索性能？**

A:
1. **索引优化**: ChromaDB自动构建HNSW索引，加速近邻搜索
2. **批量查询**: 多个query合并为一次请求
3. **缓存**: 对高频query缓存检索结果（Redis）
4. **分库**: 按业务领域拆分知识库，减少单次检索范围

**Q: 如何优化前端加载速度？**

A:
1. **代码分割**: Vite自动按路由分割chunk
2. **懒加载**: 路由组件使用`() => import()`动态导入
3. **CDN**: 静态资源（Element Plus、ECharts）使用CDN
4. **压缩**: Vite自动Tree-shaking和代码压缩

---

### 6.4 安全性问题

**Q: 如何保护API安全？**

A:
1. **JWT认证**: 用户登录后颁发token，每次请求携带
2. **CORS**: FastAPI配置允许的前端域名
3. **Rate Limiting**: 限制单IP请求频率（防爬虫）
4. **敏感信息**: API Key存环境变量，不提交代码仓库

**Q: 如何防止SQL注入/XSS攻击？**

A:
1. **无SQL**: 使用ChromaDB（NoSQL）和JSON文件，无SQL注入风险
2. **参数验证**: Pydantic自动验证类型和格式
3. **XSS防护**: Vue自动转义HTML，Markdown渲染使用DOMPurify清洗
